<NewsItem
  title="Composer 2.5 : le modèle spécialisé tient tête à la frontière"
  category="ia"
  tags={["cursor", "modèle", "coût"]}
  sources={[{ label: "Cursor", url: "https://cursor.com/blog/composer-2-5" }, { label: "via AlphaSignal", url: "https://alphasignal.ai" }]}
>
[Composer 2.5](https://cursor.com/blog/composer-2-5) (Cursor, bâti sur [Kimi K2.5](https://huggingface.co/moonshotai/Kimi-K2.5)) score **62** sur le [Coding Agent Index](https://artificialanalysis.ai/agents/coding-agents) d'Artificial Analysis, à 4 points d'Opus 4.7 (66) et GPT-5.5 (65). Mais à **0,07 $** la tâche contre ~4 $ pour les modèles frontière, soit jusqu'à 60× moins cher : 0,50 $/2,50 $ le million de tokens entrée/sortie, contre ~5 $/25 $. Cursor a consacré ~85 % de son budget de calcul au RL et aux données synthétiques (×25 de tâches vs Composer 2).
</NewsItem>

<NewsItem
  title="L'argument économique des modèles verticaux"
  category="ia"
  tags={["inférence", "fine-tuning"]}
  sources={[{ label: "Thinking Machines", url: "https://thinkingmachines.ai/news/announcing-tinker/" }, { label: "via AlphaSignal", url: "https://alphasignal.ai" }]}
>
Les boucles agentiques tournant en continu sur GPT-5.5 ou Opus 4.7 épuisent les budgets d'API avant de générer du ROI. La parade : router 80–90 % du code vers un modèle efficace, réserver la frontière aux 10–20 % de cas complexes. L'analyse cite [Tinker](https://thinkingmachines.ai/news/announcing-tinker/) (Thinking Machines), API Python de fine-tuning distribué, comme brique pour que d'autres équipes rejouent la stratégie de Cursor sur leurs propres données.
</NewsItem>
